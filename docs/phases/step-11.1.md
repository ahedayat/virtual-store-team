# Step 11.1 — React Frontend Migration to Next.js

## Objective

Migrate the existing React frontend generated by Magic Patterns into a professional Next.js application without changing the visual appearance.

The migrated application must reproduce the Magic Patterns Preview as closely as possible while adopting a scalable architecture suitable for future backend integration.

---

## Scope

This phase only focuses on frontend migration.

Backend integration is intentionally postponed.

Mock data may continue to be used where necessary.

---

## Migration Goals

- Preserve the original UI.

- Preserve component behavior.

- Preserve responsive layouts.

- Preserve animations.

- Preserve typography.

- Preserve spacing.

- Preserve colors.

- Preserve icons.

- Preserve overall user experience.

---

## Target Stack

- Next.js (App Router)

- TypeScript

- CSS Modules

- Axios

- React Query

- Zustand

- React Hook Form

- Zod

---

## Project Structure

The migrated application should follow a clean and modular architecture.

Example structure:

```

app/

components/

features/

hooks/

services/

stores/

styles/

types/

utils/

```

---

## State Management

Global state management should use Zustand.

Stores should be organized inside the `stores` directory.

Components must subscribe only to the required slices of state to minimize unnecessary re-renders.

Development builds should enable `simple-zustand-devtools`.

---

## Form Validation

All user input must be validated using React Hook Form and Zod.

Validation rules include:

- Mobile numbers must be strings beginning with `09`.

- OTP values must contain exactly six numeric digits.

---

## API Architecture

Although backend integration is outside the scope of this phase, the project should already contain a reusable API abstraction.

A generic Axios-based API client should be implemented in:

```

services/api-client.ts

```

Future API communication will be implemented through React Query custom hooks.

---

## React Query

The application should include:

- QueryClient

- QueryProvider

- React Query Devtools

Future requests should use:

- useQuery

- useMutation

Proper loading and error handling should already be supported.

---

## Styling

Inline styles should be replaced with CSS Modules whenever possible.

Global CSS should remain minimal and only contain shared variables, typography, reset rules, and application-wide styles.

---

## Component Design

Large JSX blocks should be decomposed into reusable components while avoiding excessive fragmentation.

Components should each have a clear and focused responsibility.

---

## Migration Strategy

The migration should proceed incrementally.

For each page:

1. Analyze the existing React implementation.

2. Identify reusable components.

3. Extract styles into CSS Modules.

4. Rebuild the page using Next.js.

5. Verify that the rendered output matches the Magic Patterns Preview.

---

## Expected Outcome

Upon completion of this phase:

- The Next.js application visually matches the original React application.

- The project is organized according to a scalable architecture.

- Backend integration can begin in subsequent phases without major frontend refactoring.

- The codebase follows modern frontend engineering best practices and is ready for future expansion.
